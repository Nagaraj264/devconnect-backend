QUESTION

How to implement efficient caching for real-time GraphQL subscriptions in high-traffic apps?

I've been working on a dashboard that handles thousands of concurrent users via Apollo Subscriptions. We're noticing a significant latency spike when multiple events trigger simultaneously. What's the best strategy for caching these updates at the edge?

Currently, our resolver looks something like this:

subscription OnMetricUpdate($id: ID!) {
  metricUpdated(id: $id) {
    id
    value
    timestamp
    metadata {
      source
      frequency
    }
  }
}

Is anyone using Redis as an intermediate message broker to throttle these updates before they hit the client-side cache?
